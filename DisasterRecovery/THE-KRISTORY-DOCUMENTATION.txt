# THE KRISTORY — COMPLETE PROJECT DOCUMENTATION
# Disaster Recovery & Rebuild Guide
# Last updated: March 5, 2026
# ============================================================

This document contains everything needed to rebuild The Kristory
from scratch if all code and data are lost. It covers the tech
stack, architecture, database schema, feature set, design system,
deployment, data model, and dependencies.

Keep this file safe — ideally in multiple locations (cloud storage,
local backup, printed copy).


# ============================================================
# 1. WHAT IS THE KRISTORY?
# ============================================================

The Kristory is a shared daily journal PWA (Progressive Web App)
for two users — Chris and Krista. It's a digital scrapbook and
daily diary where they document their life together through
narrative entries, photos, tagged items (movies, restaurants, etc.),
trips, recipes, and memories.

URL: https://the-kristory.vercel.app


# ============================================================
# 2. TECH STACK
# ============================================================

Frontend:
  - Vite (build tool)
  - React 18+ (UI framework)
  - TypeScript (language)
  - Tailwind CSS (styling)
  - React Router v6 (routing)
  - TanStack React Query v5 (server state management)
  - Tiptap (rich text editor, built on ProseMirror)
  - date-fns (date formatting/parsing)
  - vite-plugin-pwa (PWA/service worker)

Backend/Database:
  - Supabase (PostgreSQL database + Storage + Auth)
  - Supabase JS SDK v2
  - Supabase Storage for photo uploads

Deployment:
  - Vercel (hosting, auto-deploy)
  - Domain: the-kristory.vercel.app

Environment Variables (in .env):
  - VITE_SUPABASE_URL — your Supabase project URL
  - VITE_SUPABASE_ANON_KEY — your Supabase anon/public key

The app shares a Supabase instance with other apps (Axiom Tasks,
Stint, Pulse). All Kristory tables are standalone — no table name
prefixing needed because they have unique names.

# 4. COMPLETE DATABASE SCHEMA
# ============================================================

ALL tables have RLS (Row Level Security) DISABLED.
The app is for 2 trusted users — no auth, no RLS.

CRITICAL: All Supabase queries with nested selects MUST use
explicit FK hints (e.g. !entry_id, !category_id) or PostgREST
returns 400 errors.

--- SHARED (pre-existing) ---

TABLE: users
  id          UUID PK
  name        TEXT NOT NULL
  created_at  TIMESTAMPTZ DEFAULT now()

  Data:
    Chris  — 9df51388-65e8-44bf-bfad-65729744190b
    Krista — cab9d09c-0083-4ce1-aaf5-9a045d8a4315

--- JOURNAL ---

TABLE: journal_entries
  id          UUID PK DEFAULT gen_random_uuid()
  entry_date  DATE UNIQUE NOT NULL
  created_at  TIMESTAMPTZ DEFAULT now()
  updated_at  TIMESTAMPTZ DEFAULT now()

TABLE: entry_sections
  id          UUID PK DEFAULT gen_random_uuid()
  entry_id    UUID FK → journal_entries.id ON DELETE CASCADE
  user_id     UUID FK → users.id
  content     TEXT (stored as sanitized HTML)
  created_at  TIMESTAMPTZ DEFAULT now()
  updated_at  TIMESTAMPTZ DEFAULT now()
  fts         TSVECTOR GENERATED ALWAYS AS (to_tsvector('english', coalesce(content, ''))) STORED
  UNIQUE (entry_id, user_id)

  INDEX: idx_entry_sections_fts ON entry_sections USING GIN (fts)

TABLE: entry_photos
  id            UUID PK DEFAULT gen_random_uuid()
  entry_id      UUID FK → journal_entries.id ON DELETE CASCADE
  user_id       UUID FK → users.id
  storage_path  TEXT NOT NULL (format: {entry_date}/{uuid}.jpg)
  display_order INTEGER DEFAULT 0
  created_at    TIMESTAMPTZ DEFAULT now()

--- CATEGORIES & TAGGED ITEMS ---

TABLE: categories
  id          UUID PK DEFAULT gen_random_uuid()
  name        TEXT NOT NULL
  emoji       TEXT
  is_default  BOOLEAN DEFAULT false
  user_id     UUID FK → users.id (required — set to Chris's ID for defaults)
  created_at  TIMESTAMPTZ DEFAULT now()

  Default categories (seeded):
    🎬 Movies, 📺 TV Shows, 🍽 Restaurants, 🍳 Home Cooking,
    🎭 Activities, 📖 Books, ✈️ Trips, 🎵 Music/Concerts,
    🛒 Shopping, 👶 Baby

TABLE: tagged_items
  id                UUID PK DEFAULT gen_random_uuid()
  entry_id          UUID FK → journal_entries.id ON DELETE CASCADE (NULLABLE — standalone items allowed)
  category_id       UUID FK → categories.id
  user_id           UUID FK → users.id
  name              TEXT NOT NULL
  rating            INTEGER CHECK 1-5, NULLABLE
  location_name     TEXT NULLABLE
  location_lat      DECIMAL NULLABLE
  location_lng      DECIMAL NULLABLE
  location_place_id TEXT NULLABLE
  ingredients       TEXT NULLABLE (for Home Cooking recipes)
  instructions      TEXT NULLABLE (for Home Cooking recipes)
  item_date         DATE NULLABLE (when the thing happened, independent of journal entry)
  created_at        TIMESTAMPTZ DEFAULT now()

  INDEX: idx_tagged_items_name ON tagged_items USING GIN (to_tsvector('english', name))

--- RECIPE TAGS (many-to-many for Home Cooking items) ---

TABLE: recipe_tags
  id          UUID PK DEFAULT gen_random_uuid()
  name        TEXT NOT NULL UNIQUE
  emoji       TEXT
  created_at  TIMESTAMPTZ DEFAULT now()

  Seeded: 🍝 Pasta, 🥗 Salad, 🍲 Soup, 🫙 Sauce, 🥪 Sandwich,
          🍗 Protein, 🍞 Baking, 🍪 Sweets, 🥞 Pancakes,
          🍪 Cookies, 🥦 Sides, 🍳 Breakfast

TABLE: tagged_item_recipe_tags
  tagged_item_id  UUID FK → tagged_items.id ON DELETE CASCADE
  recipe_tag_id   UUID FK → recipe_tags.id ON DELETE CASCADE
  PK (tagged_item_id, recipe_tag_id)

--- PARTICIPANTS (who watched/read/visited) ---

TABLE: tagged_item_participants
  tagged_item_id  UUID FK → tagged_items.id ON DELETE CASCADE
  user_id         UUID FK → users.id ON DELETE CASCADE
  PK (tagged_item_id, user_id)

--- TRIPS ---

TABLE: trips
  id               UUID PK DEFAULT gen_random_uuid()
  title            TEXT NOT NULL
  summary          TEXT NULLABLE
  cover_photo_path TEXT NULLABLE
  start_date       DATE NOT NULL
  end_date         DATE NOT NULL
  created_at       TIMESTAMPTZ DEFAULT now()

TABLE: trip_entries
  trip_id   UUID FK → trips.id ON DELETE CASCADE
  entry_id  UUID FK → journal_entries.id ON DELETE CASCADE
  PK (trip_id, entry_id)

--- BABY ---

TABLE: baby_milestones
  id              UUID PK DEFAULT gen_random_uuid()
  entry_id        UUID FK → journal_entries.id ON DELETE SET NULL (NULLABLE)
  title           TEXT NOT NULL
  milestone_type  TEXT NOT NULL DEFAULT 'custom'
  milestone_date  DATE NOT NULL
  notes           TEXT NULLABLE
  photo_path      TEXT NULLABLE
  created_at      TIMESTAMPTZ DEFAULT now()

TABLE: baby_profile
  id            UUID PK DEFAULT gen_random_uuid()
  name          TEXT (default 'Baby')
  due_date      DATE (2026-10-02)
  birth_date    DATE NULLABLE
  birth_weight  TEXT NULLABLE
  birth_length  TEXT NULLABLE
  notes         TEXT NULLABLE
  created_at    TIMESTAMPTZ DEFAULT now()
  updated_at    TIMESTAMPTZ DEFAULT now()

--- SUPABASE STORAGE ---

Bucket: kristory-photos
  - Public access enabled
  - Storage policies allow public INSERT, SELECT, UPDATE, DELETE
    for bucket_id = 'kristory-photos'
  - Photos stored at path: {entry_date}/{uuid}.jpg
  - Client-side resize: max 1600px longest side, JPEG 85% quality

# ============================================================
# 8. KEY TECHNICAL PATTERNS
# ============================================================

SUPABASE QUERIES — FK HINTS:
  Always use explicit foreign key hints on nested selects:
  .select('*, sections:entry_sections!entry_id(*),
           photos:entry_photos!entry_id(*),
           tagged_items:tagged_items!entry_id(*,
             category:categories!category_id(*))')
  Without !fk_column hints, PostgREST returns 400 errors.

STATE MANAGEMENT:
  - React Context: current user, dark mode preference
  - TanStack React Query: all server state
  - Optimistic updates on all mutations
  - Auto-save with debounce (1.5s)

IMAGE HANDLING:
  - Client-side resize before upload
  - Max 1600px on longest side
  - JPEG quality 0.85
  - Canvas-based resize with Math.round() on dimensions
  - URL.revokeObjectURL after use (memory leak prevention)
  - Upload to Supabase Storage: kristory-photos/{date}/{uuid}.jpg

PWA:
  - vite-plugin-pwa with autoUpdate
  - skipWaiting + clientsClaim for immediate updates
  - Cache Google Fonts
  - "Check for Updates" button in Settings

DATE HANDLING:
  - date-fns for all formatting/parsing
  - Store as YYYY-MM-DD (Postgres DATE type)
  - Journal dates in user's local timezone, not UTC

RICH TEXT:
  - Tiptap editor (bold, italic, hard break)
  - Content stored as sanitized HTML
  - Displayed with colored left border per author

# 10. KNOWN ISSUES & QUIRKS
# ============================================================

- Categories table requires user_id (not nullable) — use Chris's
  UUID when seeding defaults
- entry_id on tagged_items is NULLABLE — standalone recipes and
  list items don't need a journal entry
- Storage bucket requires explicit RLS policies for uploads
  (CREATE POLICY for INSERT/SELECT/UPDATE/DELETE)
- PWA service worker can cache old versions aggressively — use
  "Check for Updates" in Settings or clear site data
- Photo upload on mobile was fixed by: adding proper error
  handling, fixing canvas dimension rounding, fixing object URL
  memory leaks, adding storage RLS policies
- Photo delete button uses "Edit mode" toggle to prevent
  accidental deletion on mobile (fat finger issue)
- The 2025 journal entries were initially imported as 2026 and
  required a SQL fix: UPDATE journal_entries SET entry_date =
  entry_date - INTERVAL '1 year' WHERE EXTRACT(YEAR FROM
  entry_date) = 2026
- Import parser needs to handle "Month Day, Year:" format with
  year extraction from the date header
- vite.svg 404 error on load is harmless (default Vite favicon
  never replaced in index.html)
